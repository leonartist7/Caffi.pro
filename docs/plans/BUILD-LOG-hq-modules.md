# Build log — HQ Control Center rewire (modular-painting-castle plan)

Tracks progress against `docs/plans/PLAN-hq-control-center.md` (see plan text in the PR/session). One bullet per phase.

## Phase 1 — Kill the slow navigation

- `app/(dashboard)/layout.tsx` and `app/(owner)/layout.tsx`: swapped the per-navigation `auth.getUser()` (network round-trip to Supabase Auth) for `auth.getSession()` (cookie-local). `middleware.ts` still runs the authoritative `getUser()` once per request and refreshes the cookie; these layouts now just read that already-validated session for the UX gate. All data access remains independently authorized via `requireVenueRole`/`requireAroAdmin` in each route — this change touches rendering gate only, not authorization.
- Verified: `npx tsc --noEmit` clean, `npm run build` green, `middleware.ts` confirmed unchanged (`getUser()` still present).

## Phase 2 — HQ overview dashboard at /dashboard

- Rewrote `app/(dashboard)/dashboard/page.tsx`: aro_admin now renders a real HQ overview instead of redirecting to `/clients` — stat tiles (clients, members, visits this week + delta, new leads), "Clients at a glance" per-venue mini list, recent activity feed off `events` (friendly type labels), quick actions to `/clients` and `/leads`. Non-admin role dispatch (owner/manager → `/home`, else → `/counter`) unchanged. Single `Promise.all` for all data, no new RPC.
- Self-review pass (`/code-review --level medium`) caught two real bugs before commit, both fixed:
  1. `members`/`visits` row-fetch queries had no `.limit()` — PostgREST's default max-rows cap would have silently truncated totals once platform scale passed ~1000 rows, with no error surfaced. Fixed by adding explicit `{ count: 'exact', head: true }` queries for the headline totals (never truncated) and bounding the row-fetches (used only for per-venue breakdown) to 5000.
  2. `venues` query had no `ORDER BY` — "Clients at a glance" would show an arbitrary/unstable subset once there were more than 8 clients. Fixed by ordering `created_at DESC`, matching `/clients`.
- Verified: `npx tsc --noEmit` clean, `npm run build` green.

## Phase 3 — Staff API + page rewire (fixes "Unknown error" on Add Staff)

- New `lib/invites.ts`: extracted the invite-creation logic shared by `/api/invites` and the new `/api/staff` POST, so both stay in lock-step with the `memberships` schema.
- New `app/api/staff/route.ts` (GET list / POST invite), `app/api/staff/[id]/route.ts` (PATCH / DELETE), `app/api/staff/[id]/pin/route.ts` (POST, delegates to the `set_counter_pin` RPC — hash never touches app code).
- Rewired `app/(dashboard)/staff/page.tsx` off the deprecated `staff_users` table (browser-direct, wrong columns, wrong roles) onto the routes above. Roles are now exactly `owner | manager | staff`; removed the `barista/kitchen/cashier` options and all `can_manage_*`/`assigned_location_id` fields that don't exist on `memberships`. Added a "Set counter PIN" action per staff row. Pending invites (`user_id IS NULL`) show as "Invited"; DELETE only revokes pending invites, accepted members are deactivated via `PATCH { is_active: false }` instead (their history — visits logged, redemptions processed — must survive).
- **Found and fixed a pre-existing, unrelated bug while wiring this**: `requireRowVenueRole` in `lib/authz.ts` hardcoded `.select('tenant_id')`, but its only prior caller (`/api/ai-drafts/[id]`) operates on `ai_drafts`, which has `venue_id`, not `tenant_id` — meaning the owner's AI-drafts Approve/Skip buttons have silently 404'd since they were built. Added an explicit `venueColumn` parameter (defaults to `'tenant_id'` so every other existing caller is unaffected) and fixed the ai-drafts route to pass `'venue_id'`. The new staff routes also pass `'venue_id'` correctly since `memberships` uses that column too.
- Self-review pass (`/code-review --level medium`) caught a real authorization gap before commit, fixed:
  - `PATCH`/`DELETE /api/staff/[id]` only re-checked the manager-only-touches-staff rule against the _requested_ role field, not the _target row's current_ role. A manager could deactivate/rename an owner (via `is_active`/`full_name` with no `role` field) or revoke a pending owner/manager invite, since neither path is gated by the target's existing role. Fixed by fetching the target row's current role first and refusing with 403 if a manager caller targets a non-staff row.
- Verified: `npx tsc --noEmit` clean, `npm run build` green, all three new routes present in the build's route manifest. Column/RPC-param names re-checked by hand against `20260707000001_aro_platform_schema.sql` (Supabase MCP was disconnected at commit time, so the live POST→GET→PIN round trip on real data is **deferred** — queued to run as soon as the connector is back, before Phase 3 is marked fully done in the task tracker).

## Phase 4 — Rewards, Activity, Analytics, Settings

- **Rewards**: new `app/api/rewards/route.ts` (+`[id]`) against the live `rewards` table (reachable only through the read-only `rewards_catalog` compat view before — writes there were blocked exactly like the original `/clients` bug, since the view only has a SELECT grant). DELETE checks for existing `redemptions` first and returns a friendly 409 instead of the raw FK-violation the DB would throw (`redemptions.reward_id` is `ON DELETE RESTRICT`).
- **Activity**: new `app/api/activity/route.ts` reads the real `events` table instead of the nonexistent `admin_activity_log`. With `venue_id`, scoped to that venue (owner/manager, aro_admin passthrough); without it, aro_admin-only cross-venue feed. Extracted `EVENT_LABELS`/`eventLabel`/new `activityAction`/`activityResourceType` helpers into `lib/events.ts` so the dashboard's recent-activity panel and this route share one label source instead of duplicating the map.
- **Analytics**: new `app/api/analytics/route.ts` — full content rewrite, not just a data-layer swap (the old page was revenue/orders/categories-shaped with no live equivalent; plan called for genuine loyalty analytics). KPI tiles: total members, visits in range, points issued/redeemed in range. Two fixed 12-venue-local-week trend charts (member growth, visits) per the plan's explicit spec, independent of the date-range selector. Status breakdown + top members sourced from `member_status`. "Most Active Day" and "Regulars Rate" insights replace the old page's hardcoded fake values ("Saturday +23%", "8-10 AM") — computed from real visit timestamps and status counts instead.
- **Settings**: "General" tab is now real — business name/logo/color/timezone for the selected client, wired to `/api/clients/[id]`. The other five tabs (Notifications, API Keys, Email Templates, Security, System Info) were 100% mocked/non-functional (fake toggles that saved nothing, hardcoded "45ms API response time") — replaced with a new shared `components/ComingSoon.tsx` (built now since Settings needed it; Phase 5's parked pages reuse the same component rather than duplicating the pattern).
- Self-review pass (`/code-review --level medium`) caught two real issues before commit, both fixed:
  1. **Design gap, not just a bug**: Settings' General tab depended on `GET /api/clients` and `PATCH /api/clients/[id]`, both of which were `aro_admin`-only — meaning a real café owner could never manage their own venue's brand profile, only the platform operator could, contradicting the explicit "same rich UI, venue-scoped" decision. Added `GET /api/clients/[id]` and changed `PATCH`'s gate from `requireAroAdmin` to `requireRowVenueRole(..., 'venue_id')` (aro_admin still passes; now the venue's own owner/manager does too). `DELETE` stays aro_admin-only on purpose — deleting a whole venue is a platform-operator action. Extracted `toTenantShape`/`ClientRow`/`CLIENT_COLUMNS` into a new `lib/clients.ts`, since Next.js route files can only export HTTP method handlers (learned this the hard way — the build failed until the shared shape moved out of `route.ts`).
  2. `/api/analytics`'s Points Issued/Redeemed are headline KPI tiles (not just chart data) computed by summing a `.limit(5000)`-bounded row fetch with no exact-count fallback — the same truncation class already fixed once in Phase 2, reintroduced here. PostgREST has no `{ count: 'exact', head: true }` equivalent for a signed SUM, and a live DB wasn't reachable this session to verify an aggregate-select workaround, so applied the lower-risk mitigation: raised the cap to 10,000 and added a `console.warn` that fires if the fetch returns exactly at the cap (detectable in logs rather than silently wrong). Flagged as a candidate for a dedicated aggregate RPC later.
- Verified: `npx tsc --noEmit` clean, `npm run build` green, `/api/clients/[id]` GET + all four new/changed routes present in the build's route manifest.

## Phase 5 — Module registry + parked pages

- New `lib/modules.ts`: single source of truth for the client-scoped nav (`MODULES` — staff/rewards/analytics/activity/settings `live`; menu/orders/coupons/notifications/locations `coming_soon`) and HQ-only nav (`HQ_ITEMS` — Dashboard/Clients/Leads, added Leads to the nav for the first time; it existed as a page but was never linked). `enabledModules(venue)` is forward infrastructure for the later "activate modules per client" step the owner described — not called anywhere yet since no venue has `features_enabled` overrides today, all `live` modules render by default.
- Parked `menu`, `orders`, `coupons`, `notifications`, `cafes` (+ their `[slug]` sub-pages) — bodies replaced with the shared `ComingSoon` component, routes kept so bookmarks/direct links don't 404.
- Deleted `app/(dashboard)/diagnostics/page.tsx` (browser-side inserts into `tenants` — a live footgun, per the plan).
- `/api/menu-items`, `/api/categories`, `/api/locations` (+`[id]` variants): found a real consumer before touching anything — `app/tenants/[id]/page.tsx`, a separate legacy route outside the (dashboard) group and outside this plan's scope, still calls all three. Converted them to explicit 501 `{error:'module not enabled'}` stubs instead of deleting, per the plan's own fallback for this exact situation.
- `components/Sidebar.tsx` + `components/MobileNav.tsx`: rebuilt from `lib/modules.ts` instead of a hardcoded list. HQ items (Dashboard/Clients/Leads) only render for `aro_admin`; `coming_soon` modules render with a "Soon" badge instead of being hidden. Plumbed `isAroAdmin` down from the server layout (`app/(dashboard)/layout.tsx` → `layout-client.tsx` → `Sidebar`/`MobileNav`), since neither component had any session awareness before.
- `lib/get-tenant.ts`: migrated off the legacy `tenants`/`tenant_manifests` tables onto `venues` + `brand_kit` directly, selecting only the exact column set anon is granted on `venues` (verified against the RLS migration's `GRANT SELECT (...)` list) — keeps the parked shop PWA from erroring.
- Self-review caught one real (non-security) issue before commit: computing `isAroAdmin` in the new layout reintroduced a smaller version of the "duplicate query per navigation" problem Phase 1 fixed — `dashboard/page.tsx` already ran its own membership query for the same purpose. Fixed by extracting `isAroAdminUser(userId)` into `lib/authz.ts` wrapped in React's `cache()`, so both call sites dedupe to one query per request instead of two; `dashboard/page.tsx`'s remaining owner/manager-only query now only runs for non-admins, skipping it entirely on the platform operator's own dashboard visits.

## Phase 6 — Cleanup + guardrails

- `components/TenantSelector.tsx`: the localStorage-persisted `selectedTenant` was re-synced with fresh data when still valid, but never cleared when the client had been deleted — a stale/deleted selection stayed selected forever, feeding a dead `tenant_id` into every page under it. Fixed: `setSelectedTenant(freshTenant ?? null)` — clears on mismatch instead of silently keeping the stale value or auto-picking a different client.
- `AroEventType` completeness: verified by construction, not just by inspection — `emitEvent`'s `type` param is typed against the union, so `npx tsc --noEmit` would have failed at every phase if a call site used a type not yet added. No gaps found.
